# Product

## Register

brand

## Users

Senior developers and engineering teams who use AI coding tools (Claude Code, Cursor, Codex, Cline, OpenHands, Roo Code, internal AI agents). They write production code and care deeply about security, scalability, and reliability. They are skeptical of marketing hype and respond to technical credibility. Their context: evaluating whether this collection of skills will actually help them ship safer, more robust software. They already know AI can write code — they want to know if it can catch the things they'd miss at 2am.

## Product Purpose

A collection of deeply engineered AI engineering "skills" (system prompts) designed for production-grade software development. The GitHub repository is the core product. The website exists to explain what the skills do, demonstrate their depth, and drive developers to the repo to star, fork, use, and contribute. Success looks like: a developer reads the site, immediately understands why these skills are different from generic prompts, and clicks to GitHub.

## Brand Personality

Trustworthy, Serious, Practical. Not a startup. Not a vendor. An engineering peer who has shipped production systems and knows what breaks. Voice is direct and opinionated — the copy should sound like it was written by a senior engineer, not a content marketer.

## Anti-references

- Generic SaaS landing pages: hero-metric templates, gradient blobs, icon-above-heading feature grids, rotating testimonials, "Loved by 10,000 developers" social proof strips
- Security vendor sites: red/green threat dashboards, stock imagery of locks and shields, enterprise fear-mongering copy
- AI product marketing sites: purple gradient orbs, "powered by AI" messaging, glowing chat bubbles, "the future of X" headlines
- Overly designed agency portfolios: scroll-jacking, custom cursor effects, excessive entrance animation

## Design Principles

1. **Practice what you preach.** The site itself should demonstrate engineering quality — clean code, semantic HTML, fast load, no bloat. Developers will inspect the source.
2. **Earn trust through specificity.** Generic claims ("improves security") are worth nothing. Show a real skill name, a real problem it catches, a real failure scenario it prevents.
3. **The repo is the product.** Every section should serve the CTA: get the developer to GitHub. Don't let the website compete with the repo.
4. **Restraint is the credibility signal.** No animation for animation's sake. No visual noise. A developer audience reads over-design as a credibility deficit.
5. **Show the work, don't describe it.** A real skill snippet in the hero is worth more than three paragraphs explaining what skills are.

## Accessibility & Inclusion

WCAG AA minimum. Respect reduced motion (prefers-reduced-motion). All code/skill display areas must be readable by screen readers. Keyboard-navigable. Sufficient contrast on all text, including code blocks.
