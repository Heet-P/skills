The problem

AI coding writes the happy path. Production is the sad path.

Most agents optimise for code that runs. They rarely think about code that fails — under load, under attack, under audit. Skills make that thinking explicit.

01

Generic prompts miss specialist context.

A senior security engineer doesn't think like a senior SRE. Skills encode the mental model of each discipline — what they look for, what they distrust, what they verify twice.

02

Agents trust their own output.

Skills force adversarial review — re-reading the agent's own diff with the posture of an attacker, an auditor, or an on-call engineer at 3am.

03

Production fails at the seams.

Race conditions, retry storms, expired tokens, broken webhooks. Skills include checklists drawn from real post-mortems, not generic best practices.

The library

Eight disciplines. Eighty-four skills. Drop in as needed.

Every skill is a self-contained Markdown file. Paste it into your agent's system prompt, or load it on demand the moment its domain becomes relevant.

Operating principles

How the skills think.

Every skill is written in the voice of a practitioner. These are the convictions repeated, in different words, across the library.

"Think like a real security engineer, not a tutorial."

Reason about the system as an attacker would. Assume credentials leak, tokens get replayed, and inputs are crafted, not typed.

"Test sad paths, not just happy paths."

Every new function deserves at least three failure tests: wrong input, missing dependency, partial success. The happy path is the easy one.

"Never trust the browser as a security boundary."

Client validation is UX. Server validation is the contract. Authorisation is checked on every read, every write, every join.

"Production-grade means recoverable, not perfect."

Assume something will break. Skills emphasise observability, graceful degradation, and reversible deploys over heroic correctness.

Real scenarios

Bugs an agent shouldn't have shipped — caught by a skill.

A small sample of the failure patterns the library is engineered to surface before code reaches review, let alone production.

SEC-014

IDOR hiding behind a UUID.

The agent shipped GET /invoices/:id using a UUID and assumed "unguessable" was equivalent to "authorised". The Security Audit skill re-reads every route and flags any handler that loads a resource without checking it belongs to the calling tenant.
caught by security-audit · authz-review
REL-022

Retry storms on a transient 5xx.

A naive retry: 5 with no backoff and no jitter turned a 30-second upstream blip into a self-inflicted DDoS. The Reliability skill expects exponential backoff, a circuit breaker, and an explicit budget on every outbound call.
caught by reliability · sre-readiness
SCALE-007

N+1 dressed up as "clean code."

A list endpoint mapped over results and called user.org.plan inside the loop — fine on the seed database, lethal on a real one. The Scalability skill instruments every list handler with an expected-query-count budget.
caught by scalability · db-readiness
PAY-003

Webhook trusted without signature verification.

A payment webhook was parsed and credited without checking the provider's HMAC signature. The Payments & Fraud skill refuses to consider a webhook "handled" until the signature is verified, the body is treated as raw bytes, and the event ID is checked for replay.
caught by payments-fraud · webhooks
SUPPLY-011

A "small helper" with 412 transitive dependencies.

The agent added a single utility import that pulled in a deprecated maintainer's package graph. The Dependency & Supply Chain skill checks every new dependency against age, maintainer count, install scripts and known advisory feeds.
caught by supply-chain · dep-audit
Using a skill

Drop one in. Or all of them.

Skills are plain Markdown — no runtime, no SDK. Load them as system prompts, slash commands, or as files your agent reads on demand.

~/myapp · claude code
$ gh repo clone open-skills/skills .claude/skills
cloned 84 skills · 8 categories

$ claude --skill security-audit "review the auth changes in this PR"
loading skill: security-audit.md (4.2k tokens)
scanning: src/auth/* · src/api/middleware/*

⚠ src/api/invoices.ts:42 — resource loaded without tenant scope check
⚠ src/auth/session.ts:88 — session token compared with == (timing leak)
✗ src/api/webhooks.ts:14 — webhook handler does not verify signature
✓ 3 findings · suggested patches written to .claude/review.md

$
i.

Clone the repo into your project.

Drop the skills/ folder into .claude/, .cursor/, or wherever your agent looks for prompts. One file per skill, organised by discipline.

ii.

Invoke on demand, or always-on.

Use --skill <name> for focused passes — a security review on a PR, a scalability sweep on a new endpoint — or include core skills in your default system prompt for every session.

iii.

Compose, fork, contribute.

Skills are designed to be edited. Fork your own variants, combine two skills into a domain-specific one, and open a PR when you find a failure mode the library doesn't catch yet.
